# 【1】プロダクト理解
## 現在の機能
リズムガイド付きのスクワット記録をコアとし、ボスバトル、モンスター図鑑、冒険モード（すごろく）、武器収集、クラス・称号システムなど、多様なゲーミフィケーション要素を備えたWebアプリケーション。デイリーミッションやビンゴ、スマートルーチンによる目標設定、ストリーク（連続記録）管理により継続をサポートします。

## 想定ユーザーとユースケース
「運動を習慣化したいが、モチベーションの維持が苦手な人」がメインターゲット。毎日数分、ゲーム感覚でスクワットを実施し、自身の成長やコレクションの達成感を得るために利用されます。

## 直近の変更履歴
音声ペルソナ機能の追加など、ユーザーへのフィードバックやインタラクションの質を高めるアップデートが行われています。

# 【2】アイデア創出
以下の機能案を検討しました。

1. **フェニックス・プロトコル（ストリーク復活クエスト）**
   連続記録が途切れた際、通常の2倍のスクワットなどを課す「特別クエスト」を提示。その日のうちにクリアすれば途切れた前日分の記録が補填され、ストリークが復活する機能。
2. **スマート・スタート（究極のワンタップ準備）**
   ミッション達成に必要な回数、最適な装備、おみくじの獲得などをすべて全自動でセットし、ワンタップで即座にトレーニングを開始できるボタンの追加。
3. **適応型難易度スケーリング（オートアジャスト）**
   直近のクリアタイムや連続達成日数に基づき、デイリーミッションの要求回数やボスのHPを自動調整し、常に「少し頑張れば達成できる」適切な負荷を維持する機能。
4. **トレーニング中断・自動復帰機能**
   誤ってブラウザを閉じたりリロードした場合に、現在のセット・回数・ボスHPをlocalStorageから復元し、途中から再開できるフェイルセーフ機能。

# 【3】機能選定
**選定機能：フェニックス・プロトコル（ストリーク復活クエスト）**

- **選定理由**:
  - **ユーザー価値**: 習慣化アプリにおける最大の離脱原因は「1日休んで連続記録が途切れた時の自己嫌悪とモチベーション喪失」です。このネガティブな体験を「挽回可能な熱いイベント」に変換することで、劇的に継続率を向上させます。
  - **差別化要素**: 課金やアイテム消費による安易な復活ではなく、「自らのスクワット（努力）」で復活を勝ち取るという、フィットネスアプリの本質を突いたアプローチです。
  - **実装難易度/技術的リスク**: 既存の `StreakGuardian` と履歴保存ロジック（`DataManager`）の間に介入するだけで実現可能であり、現実的なコストと低いリスクで高い効果を生みます。

# 【4】仕様策定 (SDD)
## 目的
記録が途切れてしまったユーザーに挽回のチャンスを与え、挫折によるアプリ離脱を防ぐ。

## 機能要件
- `StreakGuardian` が `lost` 状態になった際、未実施日数が「1日（前日のみ）」の場合に限り、「フェニックス・プロトコル（復活クエスト）」の開始ボタンを表示する。
- 復活クエストを受諾すると、専用の目標回数（例: 50回）が設定される。
- クエストを受諾したその日のうちに目標回数を達成すると、前日の日付でダミーの「リカバリー完了」履歴データが挿入され、ストリークが復活する。
- 復活後、画面上に派手な祝福演出とToastメッセージを表示する。

## 非機能要件
- 既存の履歴データ（LocalStorage: `squat-tracker-history-v1`）の構造を壊さず、ソート順を維持して安全にデータを挿入すること。

## 想定ユースケース
仕事が忙しく1日スクワットを休んでしまい、翌日アプリを開くと「記録が途切れました」と表示された。しかし「特別クエスト（スクワット50回）をクリアすれば記録を修復できる」という提案を受け、ユーザーは奮起してスクワットを実行し、見事ストリークを取り戻す。

## 入出力仕様
- **入力**: 復活クエスト受諾時のフラグ保存、ワークアウト完了時の実施回数（`totalReps`）。
- **出力**: 条件達成時に生成される前日分のダミー履歴オブジェクト、`StreakGuardian` の状態再計算。

## エッジケース
- **すでに2日以上休んでいる場合**: 挽回不可とし、復活クエストは提示しない。
- **クエスト受諾中に日を跨いだ場合**: 時間切れとして復活失敗。通常の新しいストリーク1日目として扱う。

## 失敗時の挙動
目標回数に達せずに終了した場合や、復活クエストを受諾しなかった場合は、通常の仕様通り「ストリーク1日目」として新しい記録が始まる。

# 【5】設計
## コンポーネント構成
1. `js/modules/phoenix-protocol.js` (新規)
   - 復活クエストの状態管理（LocalStorage `squat-tracker-phoenix`）、受諾処理、達成判定、履歴修復ロジックをカプセル化。
2. `js/modules/streak-guardian.js` (改修)
   - `lost` 判定時に「1日遅れか」を評価し、該当する場合は PhoenixProtocol のUI（復活ボタン）をレンダリングする。
3. `js/app.js` (改修)
   - ワークアウト完了時のイベントフロー内で `PhoenixProtocol.checkCompletion(totalReps)` を呼び出し、達成演出と状態リセットを行う。

## データ構造
LocalStorage Key: `squat-tracker-phoenix`
```json
{
  "isActive": true,
  "targetReps": 50,
  "missedDate": "2023-10-01"
}
```

## 責務分離
- **PhoenixProtocol**: 「復活」というビジネスロジックの提供と状態管理。
- **StreakGuardian**: ストリーク状態の監視とUI表示。
- **DataManager**: 履歴配列への安全な追加処理（必要に応じて既存メソッドを利用または拡張）。

## テスト方針
- Playwright を用いて、履歴データを意図的に「1日抜け」の状態にしたモックを用意し、アプリを起動。
- StreakGuardian に復活ボタンが表示されることを確認。
- ボタンを押下後、指定回数のスクワット（モック上での完了処理）を行い、ストリークが繋がってUIが「safe」や「completed」に戻ることを検証する。
