# インターバル・メディテーション (Rest Meditation)

## 【1】プロダクト理解
- **現在の機能**: スクワットを記録するアプリにRPG要素（ボス討伐、武器・バディ、クラス、称号）を組み合わせ、継続をサポートする。音声コーチやヒートマップ、デイリーミッションなどのモチベーション維持機能が充実している。
- **想定ユーザーとユースケース**: 筋トレが続かない層や、ゲーム感覚で毎日の成長を感じたいユーザー。主に自宅でスマホを使いながら、1日数分〜数十分のスクワットを行う。
- **既存の課題**: トレーニング機能は充実しているが、セット間の「休憩時間（REST）」は待つだけの時間になりがちであり、ここでスマホの別アプリを開いてしまい離脱する（集中が切れる）リスクがある。

## 【2】アイデア創出
1. **フィットネス・ガチャ (Squat Gacha)**: 運動で得たコインで武器やバディを引くランダム要素。
2. **インターバル・メディテーション (Rest Meditation)**: 休憩時間に呼吸を整えるアクションをゲーム化し、次セットのボーナスを得る。
3. **トレーニング日記 (Workout Journal)**: 終了後に体調やメモを残し、ヒートマップで振り返る。
4. **デイリー・クエストボード (Guild Requests)**: 毎日ランダムなNPCから特定のトレーニング条件の依頼が来る。

## 【3】機能選定
**選定機能: インターバル・メディテーション (Rest Meditation)**
- **ユーザー価値**: 休憩中もアプリへの没入感を維持し、実際のトレーニングにも役立つ「深い呼吸」を促す。ゲーム的には次セットのバフを得るための「チャージ行動」として機能する。
- **差別化要素**: 単なるインターバルタイマーではなく、RPGの「瞑想・溜め」のアクションとして体験を再構築している点。
- **実装難易度**: 既存のフェーズ管理（Phase.REST）に合わせてUIを表示し、タップイベントを処理するだけでよく、実装コストが現実的。
- **技術的リスク**: コアの戦闘やタイマーロジックを破壊するリスクが低く、安全にアドオンできる。
- **将来拡張性**: クラス（戦士なら「武器の手入れ」等）によってインターバル中のアクションや得られるバフの種類を変えるなどの拡張が可能。

## 【4】仕様策定
### 目的
休憩時間（RESTフェーズ）中のユーザー離脱を防ぎ、呼吸を整える動作をゲーム化することで、次のセットへのモチベーションとゲーム内メリットを提供する。

### 機能要件
- 休憩フェーズ（Phase.REST）中に、画面に「瞑想（Meditation）」UIがオーバーレイ表示される。
- UIには呼吸のガイドとなる円が収縮・拡大するアニメーションが表示される。
- 円が特定のサイズ（Target Ring）に重なったタイミングでユーザーが画面（または専用ボタン）をタップすると、タイミングに応じて「Perfect」「Good」などの判定が行われる。
- 成功するごとに「Focus（集中力）」ポイントが蓄積される。
- 休憩終了時（またはPhase.REST_COUNTDOWN移行時）に、蓄積されたFocusポイントに応じて、次のセットの攻撃力ボーナスやテンションゲージ増加が付与される。

### 非機能要件
- 既存のワークアウトタイマーの進行を妨げないこと。
- パフォーマンスに影響を与えないよう、CSSアニメーション（`transform`, `opacity`）を中心に描画すること。

### 想定ユースケース
1. ユーザーが1セット目のスクワット（10回）を完了する。
2. 30秒の休憩（Phase.REST）に入る。
3. 画面に瞑想の呼吸リングが表示され、「タイミングよくタップして集中力を高めろ！」とガイドが出る。
4. ユーザーは呼吸に合わせてタップし、Focusポイントを溜める。
5. 休憩が終わり2セット目が始まると、「Focus Bonus! 攻撃力+5 / テンション+10%」という通知とともにバフがかかる。

### 入出力仕様
- **入力**: 瞑想UIへのタップ/クリックイベント。
- **出力**: タップの判定結果（ポップアップテキスト）、Focusポイントの加算、休憩終了時に `sessionAttackBonus` と `TensionManager` に数値を還元。

### エッジケース
- 休憩時間が非常に短い（例: 10秒以下）設定の場合: 瞑想UIは表示されるが、リングのサイクルが速くなる、またはボーナス上限が低くなる。基本は時間に依存してタップ可能回数が決まる。
- 一時停止（Pause）が行われた場合: アニメーションを停止、または非表示にする（今回は簡単のため一時停止中はアニメーションも一時停止状態にするか、そのままにする）。
- 自動進行: ユーザーが一切タップしなくてもエラーにはならず、ボーナスが0になるだけ。

### 失敗時の挙動
- ペナルティは発生しない。

## 【5】設計
### コンポーネント構成
- `js/modules/meditation-system.js`: 瞑想のロジック、ポイント計算、タイミング判定を管理する独立したモジュール。
- `index.html`: `.workout-card` 内部に `<div id="meditation-container">` を追加。
- `styles.css`: 呼吸を示すアニメーション（`@keyframes breathe`）、判定テキストのスタイル。

### データ構造
```javascript
const state = {
  isActive: false,
  focusPoints: 0,
  maxRadius: 100,
  currentRadius: 0
};
```

### API / UI 仕様
- `MeditationSystem.init()`: 初期化（DOM要素の取得とイベントリスナーのバインド）。
- `MeditationSystem.start(duration)`: 休憩開始時に呼び出し、UIを表示してアニメーションを開始。
- `MeditationSystem.stop()`: 休憩終了時に呼び出し、UIを隠す。獲得したボーナスをオブジェクト `{ attackBonus, tensionBonus }` 形式で返す。

### 責務分離
- `app.js` はフェーズ移行のトリガーと、ボーナスの適用のみを行う。
- `MeditationSystem` はミニゲームの進行と描画更新のみに専念する。

### テスト方針
- Playwrightを使用し、`tests/meditation.spec.js` を作成。
- 休憩フェーズに移行した際に `#meditation-container` が可視状態になることを確認する。
- タップイベントを発火させ、FocusポイントのUIが更新されることを確認する。
- 休憩終了後、`sessionAttackBonus` に値が反映されているかを内部変数の検証、またはダメージ出力等から間接的に確認する（あるいはToast通知のテキストを検証する）。
